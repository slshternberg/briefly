# Deploy: 0.1 — Google OAuth Token Encryption

Encrypts Google OAuth access and refresh tokens at rest using AES-256-GCM.

## Prerequisites

- `ENCRYPTION_KEY` env var set to a 64-char hex string (32 bytes).
  Generate one: `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`
- Ensure it is in `.env.production` (or your secrets manager) before deploying.

## Deployment sequence

### Step 1 — Deploy the code

Deploy the build that includes:
- `prisma/migrations/20260421000000_add_encrypted_google_token_columns`
- `src/services/google/tokens.ts`
- Updated `callback/route.ts`, `gmail/index.ts`, `send-email/route.ts`

Run on the server:
```bash
npx prisma migrate deploy
```

This adds `googleAccessTokenEncrypted` and `googleRefreshTokenEncrypted` columns.
The old plaintext columns (`googleAccessToken`, `googleRefreshToken`) still exist.

At this point the app dual-writes: new users get encrypted tokens immediately.
Existing users still read from plaintext columns (fallback path).

### Step 2 — Run backfill

Dry-run first to count affected rows:
```bash
DRY_RUN=true npx tsx src/scripts/backfill/encrypt-google-tokens.ts
```

Then run for real:
```bash
npx tsx src/scripts/backfill/encrypt-google-tokens.ts
```

Verify zero users remain with plaintext-only tokens:
```sql
SELECT COUNT(*) FROM users
WHERE "googleAccessToken" IS NOT NULL
  AND "googleAccessTokenEncrypted" IS NULL;
-- Must return 0
```

### Step 3 — Wait 24 hours

Monitor for errors. If rollback is needed, the previous deploy still works because
plaintext columns are still present.

### Step 3.5 — Deploy cleanup code (before Step 4)

Before dropping the plaintext columns, deploy a version of `tokens.ts` that:
- Removes `googleAccessToken` and `googleRefreshToken` from the `select` in `getDecryptedGoogleTokens`
- Removes the plaintext fallback branch
- Removes dual-write calls in `setEncryptedGoogleTokens` and `refreshEncryptedAccessToken`

This prevents the running app from querying columns that are about to be dropped.
Without this step, Step 4 will immediately cause "column does not exist" errors on
every Gmail read — even though the fallback path would never actually execute.

Only after this cleanup deploy is live and confirmed → proceed to Step 4.

### Step 4 — Drop plaintext columns

Only after Step 3.5 is live and confirmed clean:
```bash
npx prisma migrate deploy
# applies: 20260421000001_drop_plaintext_google_tokens
```

This migration is guarded with a comment block — do not run it early.

## Rollback

Before Step 4: redeploy the previous build. Old code reads plaintext columns which
still exist. No data loss.

After Step 4: plaintext columns are gone. Rollback requires restoring from backup
or re-running `ALTER TABLE users ADD COLUMN "googleAccessToken" TEXT` and
re-populating from the encrypted columns (decrypt + insert).

## Verification

After Step 1 + Step 2, confirm:
1. `googleAccessTokenEncrypted` is non-null for all users with `googleAccessToken`
2. A test Gmail send works end-to-end
3. Token refresh (7-day expiry) works (check logs for "Failed to persist refreshed OAuth token" errors — there should be none)
